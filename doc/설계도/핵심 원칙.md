### 1-1. 레이어 구조

- **Pages (View)**: 라우트 단위 화면, 컴포넌트 조합만 담당
- **Components (UI)**: Base / Domain / Layout 3계층
- **Pinia Stores / Composables**: 전역 상태 / 재사용 로직
- **Supabase Client**: Auth · DB 호출 단일 창구 (`@nuxtjs/supabase`)
- **Supabase (Cloud)**: Postgres · Auth · RLS

| 원칙 | 내용 |
| --- | --- |
| 서버는 얇게 | Nuxt 서버는 미들웨어(세션 확인)만 담당. CRUD는 전부 클라이언트에서 직접 호출 |
| RLS가 1차 보안 | DB 레벨 `auth.uid() = user_id` 강제. 클라이언트 필터는 보조 수단 |
| 단방향 데이터 흐름 | `Supabase → Store → Page → Component` 방향만 허용 |
| 컴포넌트 역할 분리 | Base(토큰 소비만) → Domain(비즈니스 UI) → Page(조합) 3계층 엄수 |
| Composition API 전용 | Options API 사용 금지. `<script setup>` 문법 통일 |
| 전역 상태 최소화 | `auth.store`  • `diary.store` 2개만 유지. 나머지는 Composable 또는 Page 로컬 상태 |
| Store만 Supabase 호출 | Page·Component에서 직접 supabase 호출 금지. 반드시 Store 액션 경유 |
