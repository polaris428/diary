# 프로젝트 구조

> 기준 문서: `doc/설계도/아키텍처.md`

## 1. 목표 구조

Nuxt에서는 안드로이드처럼 빌드 모듈을 강제하지 않는다.

대신 아래 4개 경계로 프로젝트를 나눈다.

- `app`: Nuxt 엔트리
- `features`: 기능 구현
- `shared`: 공통 UI/유틸/타입
- `core`: 앱 전역 인프라

권장 구조 예시는 아래다.

```text
app/
  pages/
  layouts/
  middleware/

features/
  auth/
    components/
    composables/
    store/
    types/
  diary-list/
    components/
    composables/
    store/
    types/
  diary-detail/
    components/
    composables/
    store/
    types/
  diary-write/
    components/
    composables/
    store/
    types/
  settings/
    components/
    composables/
    store/
    types/

shared/
  components/
    base/
    layout/
  composables/
  utils/
  types/

core/
  auth/
  supabase/
  error/
  session/
```

## 2. 디렉토리 역할

### `app/`

- Nuxt 페이지 진입점
- 레이아웃
- 전역 미들웨어
- 기능 조합만 담당

여기서는 하지 않는 것:

- 기능 CRUD 구현
- 공통 유틸 정의
- 기능 전용 컴포넌트 상세 구현

### `features/`

- 사용자 기능 단위 구현
- 한 기능을 수정할 때 가장 먼저 읽는 영역
- 기능 전용 상태, UI, 타입, composable을 포함

예:

- 로그인: `features/auth`
- 일기 목록: `features/diary-list`
- 일기 상세: `features/diary-detail`
- 일기 작성: `features/diary-write`

### `shared/`

- 여러 feature에서 재사용하는 코드
- 순수 UI, 순수 유틸, 공통 타입만 위치

예:

- `shared/components/base/BaseButton.vue`
- `shared/components/layout/AppHeader.vue`
- `shared/utils/date.ts`

### `core/`

- 앱 전체가 공통으로 의존하는 기반 코드
- 인증, 세션, Supabase 연결, 전역 에러 정책

예:

- `core/auth`
- `core/supabase`
- `core/error`

## 3. 의존성 규칙

허용 방향:

```text
app -> features, shared, core
features -> shared, core
core -> shared
shared -> nothing
```

금지 방향:

- feature -> 다른 feature 내부 직접 참조
- shared -> feature 참조
- core -> feature 참조

판정 원칙:

- 특정 기능 하나에만 속하면 `features`
- 여러 기능이 공통으로 쓰면 `shared`
- 앱 전역 기반이면 `core`

## 4. 현재 코드 기준 매핑

현재 저장소는 아직 Nuxt 기본 구조 중심이다.

```text
pages/
components/
stores/
composables/
middleware/
utils/
types/
```

이 구조를 아래처럼 해석한다.

- `pages/` -> `app/pages`
- `layouts/`, `middleware/` -> `app`
- `components/base/`, `components/layout/` -> `shared/components`
- `utils/`, `types/` -> `shared`
- `stores/auth.store.ts` -> 장기적으로 `features/auth/store/`
- `stores/diary.store.ts` -> 장기적으로 `features/diary-*`
- `components/diary/` -> 장기적으로 `features/diary-*`
- `composables/useQuestion.ts`, `composables/useToast.ts` -> 성격에 따라 `shared` 또는 feature 내부

참고:

- `core/auth`는 인증 인프라와 세션 기준을 둔다.
- 로그인 화면 흐름과 사용자 인증 기능 상태는 `features/auth`에 둔다.

## 5. 점진적 전환 원칙

현재 코드를 한 번에 전면 이동하지 않는다.

전환 순서:

1. 새 기능부터 `features` 기준으로 작성
2. 기존 공통 UI는 `shared` 해석으로 유지
3. 앱 전역 인프라는 `core`로 명시
4. 기존 루트 `stores`, `components`, `composables`는 점진적으로 기능별 위치로 이동

즉, 구조 전환은 "새 규칙 우선 적용, 기존 코드는 점진 이동" 방식으로 진행한다.
