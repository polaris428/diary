# Nuxt 아키텍처 명세

> 목적: AI와 사람이 같은 구조와 같은 판단으로 구현하도록 강제하는 기준 문서

## 1. 핵심 원칙

- 기술 스택: `Nuxt 3 + Vue 3 + Pinia + Supabase`
- 기본 CRUD 경로: `Page -> Store -> Supabase`
- Nuxt `server/api`는 기본 CRUD 경로로 사용하지 않는다.
- 보안의 1차 책임은 Supabase Auth + RLS에 둔다.
- 단방향 데이터 흐름만 허용한다.

## 2. 모듈 경계

Nuxt에서는 빌드 모듈 대신 디렉토리 경계와 import 규칙으로 모듈화를 구현한다.

- `app`: Nuxt 엔트리
- `features`: 기능 구현
- `shared`: 공통 UI/유틸/타입
- `core`: 앱 전역 인프라

의존성 규칙:

```text
app -> features, shared, core
features -> shared, core
core -> shared
shared -> nothing
```

금지 규칙:

- feature -> 다른 feature 내부 직접 참조 금지
- shared -> feature 참조 금지
- core -> feature 참조 금지

## 3. 레이어 책임

| 레이어 | 역할 | 금지 |
| --- | --- | --- |
| `app` | 라우트, 레이아웃, 미들웨어, 조합 | 기능 구현 세부 로직 소유 |
| `features` | 기능별 UI, 상태, 액션, 타입 | 다른 feature 내부 직접 참조 |
| `shared` | 공통 UI, 공통 유틸, 공통 타입 | 기능별 상태/정책 소유 |
| `core` | auth, supabase, session, error 등 전역 기반 | 기능 화면 소유 |

## 4. Nuxt 기준 책임 분리

### `pages`

- 화면 조합
- Store 액션 호출
- 로딩/에러/빈 상태 분기
- 페이지 로컬 상태
- 라우트 이동

금지:

- Supabase 직접 호출
- 공통 비즈니스 로직 중복 작성

### `stores`

- CRUD 및 외부 I/O
- 기능 상태 소유
- 성공/실패 결과 반환

금지:

- DOM 접근
- 컴포넌트 참조

규칙:

- Store만 Supabase를 호출한다.
- 기능 전용 Store는 `features/<feature>/store/`에 둔다.
- 전역 Store는 꼭 필요할 때만 둔다.

### `components`

- `shared/components/base`: 순수 UI
- `shared/components/layout`: 공통 프레임
- `features/*/components`: 기능 전용 UI

규칙:

- Component는 Props/Emit으로만 통신한다.
- Component에서 Store import 금지
- Component에서 `useSupabaseClient()` 호출 금지

### `composables`

- 반복 계산 로직
- UI 보조 로직
- 경량 반응형 훅

금지:

- 원본 비즈니스 상태 저장
- CRUD 액션 소유

배치:

- 기능 전용: `features/<feature>/composables/`
- 공통: `shared/composables/`

### `utils`

- 순수 함수만 허용
- Vue API 금지
- 사이드이펙트 금지

## 5. 파일 배치 규칙

새 코드는 아래 순서로 위치를 결정한다.

1. 라우트 진입점이면 `app/pages/`
2. 특정 기능 하나에만 속하면 `features/<feature>/`
3. 여러 기능이 공통으로 쓰면 `shared/`
4. 앱 전역 기반이면 `core/`

애매할 때 우선순위:

- `feature` vs `shared` -> `feature`
- `feature` vs `core` -> `feature`
- `store` vs `composable` -> `store`
- `composable` vs `utils` -> `utils`

## 5-1. 최신 구조 우선 원칙

- 작업 기준은 현재 저장소의 루트 구조가 아니라 이 문서의 최신 구조(`app / features / shared / core`)다.
- 기존 루트 `pages`, `components`, `stores`, `composables`, `utils`, `types`는 레거시 구조로 간주한다.
- 새 파일은 레거시 경로에 추가하지 않고 최신 구조 기준 경로에 만든다.
- 레거시 파일은 "참고용 진실"이 아니라 "이동 대상 구현"으로 본다.

## 5-2. 레거시 수정 규칙

- 작업 대상 기능에 레거시 구조가 남아 있으면, 기능 구현과 함께 최신 구조로 옮기면서 수정한다.
- 단순 복붙 이동은 금지한다. 이동 후 책임이 맞지 않는 코드는 레이어 규칙에 맞게 분리한다.
- 레거시 파일 전체를 한 번에 전면 이동하지 않는다. 현재 작업 범위와 직접 관련된 파일까지만 이동한다.
- 레거시 코드가 남아 있어도 새 로직은 레거시 경로에 추가하지 않는다.
- 불가피하게 레거시 파일을 임시 수정했다면, 왜 최신 구조로 바로 옮기지 못했는지 문서 또는 작업 로그에 남긴다.

예시:

- 일기 작성 기능 작업 시 기존 `stores/diary.store.ts`를 계속 확장하지 말고, 관련 로직을 `features/diary-write` 또는 해당 책임을 가진 최신 경로로 옮기며 수정한다.
- 공통 버튼 수정 시 `components/base`를 계속 늘리는 대신 `shared/components/base` 기준으로 옮기며 수정한다.

## 6. 서버 계층 예외 조건

아래 경우에만 `Nuxt server/api` 또는 Function을 도입한다.

- 비밀 키 또는 서비스 키가 필요한 작업
- 외부 API 서버 중계가 필요한 작업
- 트랜잭션이 필요한 작업
- 웹훅 검증 같은 서버 전용 작업
- 여러 클라이언트가 같은 비즈니스 규칙을 공유해야 하는 작업
- RLS만으로 표현하기 어려운 권한 규칙

규칙:

- 전체 CRUD를 한 번에 서버로 옮기지 않는다.
- 필요한 액션만 선택적으로 승격한다.
- 승격 전 설계도 문서를 먼저 수정한다.

## 7. 상태 관리 규칙

- 전역 상태는 최소화한다.
- 로그인 사용자, 세션, 현재 목록/상세 정도만 전역 후보로 본다.
- 폼 입력값, 다이얼로그 열림 여부, 수정 모드는 페이지 로컬 상태로 둔다.

## 8. 네이밍 규칙

- Store: `{domain}.store.ts`
- Composable: `use{Name}.ts`
- Base Component: `Base{Name}.vue`
- Feature Component: `{Domain}{Name}.vue`
- Boolean: `is`, `has`, `can`, `should` 접두사 사용

## 9. AI 작업 규칙

- 새 기능은 `features` 기준으로 작성한다.
- `pages`는 조합만 담당한다.
- shared는 진짜 공통일 때만 사용한다.
- 구조 변경이 필요하면 구현보다 설계도 수정을 먼저 한다.
- 레거시 구조를 발견하면 그대로 확장하지 말고, 현재 작업 범위 안에서 최신 구조로 옮기며 수정한다.
